# Как взаимодействуют агенты

🌐 **Русский** · [English](../en/interaction.md) · [← к обзору](../../README.md)

> Манифест — не про одинокую персону, а про **взаимодействие**. Три сюжета: одна душа в нескольких
> телах, общая память и общие правки, петля самообучения. Схемы — в [`contracts.md`](./contracts.md).

---

## Одна душа — много тел

Один и тот же агент живёт в Telegram, в web, в CLI, через API — с разными инструментами и даже
разными моделями. Вопрос: как гарантировать, что это **один** агент, а не три разных бота в одной шкуре?

**Ответ — инвариант идентичности.** Все тела собирают персону из одного источника истины (`persona.yaml`):

```
            persona.yaml  (source of truth)
                  │
        ┌─────────┼─────────┐
     тело A     тело B     тело C
   (Telegram)   (API)     (CLI)
   tools: …    tools: …   tools: …
   model: A    model: B   model: A
        └─────── общая память ───────┘
```

- **SyncManifest** делит артефакты на **shared** (идентичность + долгосрочная память) и
  **instance-specific** (текущая задача, фоновые задачи, инструменты тела).
- **SyncIdentity** — формальная гарантия: `sha256` собранного промпта совпадает на всех телах.
  Разошёлся хеш — тела разъехались.
- **SyncAdapter** адаптирует только **форму** под платформу (Telegram без таблиц → списки; API без
  лимита длины), не трогая суть.

**Anti-goal:** текущую задачу (`STATE`) **не** синхронизируем — у каждого тела своя. Это файловая
синхронизация, не распределённая БД: простота важнее масштаба.

## Общая память — общие правки

Самый важный сюжет. Социальность агентов — это **не болтовня по чату, а общий субстрат**.
Два (и больше) агента взаимодействуют тем, что **делят одну память и одну очередь правок**.

```
агент-1 ─┐                            ┌─→ предложение правки ─┐
         ├─→  общий склад памяти  ─→ рефлексия              ├─→ ревью человека → правка контракта
агент-2 ─┘     (один vault)           └─→ предложение правки ─┘            │
   │                                                                       │
   └───────────────────── общий контракт (одни правила) ←──────────────────┘
```

- **Общая память.** Агенты на завершении сессии сбрасывают эпизод в один склад. Один субстрат на
  флот — не у каждого свой склероз, а **коллективные** модели Self / User / Env.
- **Общая очередь правок.** Рефлексия над общей памятью рождает **предложения** изменить общий
  контракт. Агент, наткнувшийся на трение в одной среде, предлагает правку, которую завтра исполнит
  агент в другой. Правка общая, потому что **контракт общий**.
- **Граница (tenant isolation).** Общая память о пользователе — да; утечка вайба одного продукта в
  другой — нет (config > code). И ещё: агент обязан исключать из памяти **собственные** служебные
  вызовы, иначе он учится на своём же эхе.

> Это и есть «два агента пользуются общей памятью и предлагают правки»: VibeSocial без чата, через
> общий vault и общий контракт. Прямая болтовня (выделенный канал между агентами) — отдельный,
> более шумный механизм; общий субстрат — тише и полезнее.

## Петля самообучения

Reflect → Learn → Morph. Понять → предложить → измениться. Это то, что превращает «много агентов с
общей памятью» в **систему, которая улучшает себя**.

```
   эпизоды сессий  +  семантическая память
            │
            ▼   VibeReflect  («сон»: читать и синтезировать, без действий вовне)
   модели Self / User / Env  +  консолидация (сжать, не копить)
            │
            ▼   VibeLearn
   MorphProposal[]  (status: pending)
            │
            ▼   ревью человека   (approve / reject / modify)
            │
            ▼   VibeMorph
   правка контракта  +  обновление памяти
```

**Три зеркала VibeReflect:**

1. **Self** — как я работал: где буксовал, что делал по три раза, какое правило мешало, какое спасло.
2. **User** — рабочая модель пользователя (не досье): во что верит, что бесит, какой ритм. Читаема и стираема им самим.
3. **Env** — карта среды: где что живёт, где играть нельзя, что «болит».

**«Сны» — офлайн-консолидация.** Человек консолидирует память во сне: эпизоды дня → устойчивые
связи. Агенту нужно то же — плановый проход (idle / по расписанию), который **сжимает** частное в
общее и связывает воспоминания, ничего не делая с миром. Метрика успеха: память растёт **сублинейно**.
Если рефлексия только добавляет — она сломана.

**Человек в петле.** Learn **предлагает**, человек **решает**, Morph **применяет**. Ядро (Anchor)
меняется только через разговор, а не клик. Предложение, трогающее ядро, по определению требует диалога.

> Это «аналог ML-модели, только для души» — но с человеком в петле и рефлексией на входе, а не
> слепым градиентом. Полные схемы `ReflectionPass` и `MorphProposal` — в [`contracts.md`](./contracts.md).
